# Interaction-lag baseline — 2026-06-26

**Pass type:** measure-and-localise only. No application code, migrations, indexes, or RLS were changed. Nothing was deployed (and would not be — it is a Friday). The single artefact of this pass is this document.

**Governing question:** the operator reports that a click takes a noticeable moment to respond, the app does not feel smooth, and the lag persists across different networks and on mobile data. Four structurally different causes present identically; this pass identifies the one dominant cost before any fix is designed.

---

## 1. Executive summary

**Dominant bottleneck class: per-interaction server round-trip.**
**Root cause: an infrastructure region mismatch — the Vercel functions run in `iad1` (Washington D.C., US East), while the database is in Sydney (`ap-southeast-2`) and the users are in Australia.**

Every authenticated page is `force-dynamic` (server-rendered on every request, no caching), and every page/action runs as a serverless function in the US. That function then reaches *back across the Pacific* to the Sydney database for each query it makes — and the interaction-heavy pages make several queries in sequence. For an AU end-user, everything in this loop is ~5–25 ms apart **except** the compute, which is ~200 ms away from *both* the user and the data — the user reaches the US for compute, and the compute reaches Sydney for data. (The operator is **currently in the US**, so right now their *user→compute* hop is fast; but the compute is still ~200 ms from the Sydney data, so the lag persists while travelling, just with a different split.)

This was measured directly, not inferred:

| Measurement (production, warm) | Result | What it isolates |
|---|---|---|
| `x-vercel-id` region on every dynamic response | `iad1` (US East) | Where compute runs |
| `/login` TTFB (function, ~no DB) | **0.14–0.25 s** | Compute hop — *US-local* (probes run from the operator's US machine; see note) |
| `/i/<uuid>` TTFB (function **+** Sydney query) | **0.79–1.0 s** (1.7 s colder) | US → Sydney query tax (~0.6–0.85 s for a *light* query set) |
| Direct TCP to Sydney AWS vs `us-east-1` | **~225 ms** vs **~15 ms** | This machine is in the US; DB is in Sydney |

**Probe-location note (added after the operator flagged travel).** I confirmed this machine is **in the US, beside `iad1`** (direct TCP ~225 ms to Sydney AWS, ~15 ms to `us-east-1`). So these probes reflect the **operator's *current* experience while travelling** (US user → US compute → Sydney data), and the ~0.15 s floor is *US-local* compute, not an AU→US hop. The robust, location-independent figure is the **~0.65–0.85 s DB tax** — the same-machine delta between the DB and no-DB routes, where the requester's location cancels out. For the **AU end-users and the operator once back in Sydney**, add a further AU→`iad1` user hop (~0.2 s) *on top*, so the permanent audience has it **worse** than the numbers here, not better.

A real authenticated surface does **5–6 sequential** Sydney round-trips per load (see §6), so the measured ~0.65–0.85 s tax on a *single light* query path extrapolates to **~1.3–1.7 s of pure network latency per portal navigation**, warm (from the US; ~0.2 s more for an AU user). That matches every symptom: it is per-interaction (not first-click-only), and it is independent of the user's own bandwidth (the killer latency is server↔data and user↔compute, not download size) — which is exactly why it shows up identically on Wi-Fi and on mobile data.

**Smallest fix that addresses the root cause (recommended, NOT implemented this pass): move the Vercel functions to the Sydney region (`syd1`) to co-locate compute with the database and the users.** This collapses each US→Sydney query round-trip from ~200 ms to ~1–5 ms and the AU→US user hop from ~150 ms to ~10–20 ms. Expected reduction in per-interaction latency: roughly 70–90%, with no application-code change. It requires a redeploy (so: not today) and a Vercel plan that permits region selection (to confirm — see §9).

**Confidence: high** for the dominant cause (measured region + measured warm latencies from both ends). The application architecture (§6) is a strong **secondary amplifier**, not the root: it multiplies one region-tax into many by issuing queries sequentially, re-verifying auth on every request, and offering no optimistic UI on the two most interactive surfaces. It becomes the leading ceiling only *after* the region is fixed.

## 3. Step 0 — Lag characterisation, per surface

The pattern alone splits the hypothesis space, so it is recorded first. What I can establish from the production probes + code, and what still needs the operator's eyes, are marked.

- **Is the worst lag the first interaction after idle, with later ones fast (→ cold start)?**
  *Partly, but it is not the main story.* I measured a real Vercel cold-start penalty (apex `/` cold = **2.19 s** vs warm ~0.16 s). **But even fully warm, a DB-touching route is ~0.8–1.0 s** — so cold start cannot be the *sole* or *dominant* cause; a steady tax is present on every warm interaction. Whether the operator perceives the *first* post-idle click as dramatically worse also depends on the Supabase plan (a free-tier project can pause/cold-start the DB) — **operator to confirm tier (§9)**. If the DB is paused on free tier, the first-after-idle click will be seconds worse on top of the steady tax.

- **Is the lag on roughly every interaction, uniformly (→ client TTI or per-interaction server round-trip)?**
  *Yes — this is the main story, and it is the server round-trip, not client TTI.* Every interaction is either a navigation (full `force-dynamic` SSR in the US + Sydney queries) or a mutation (server action in the US + Sydney RPC, most also `revalidatePath` → full re-render + refetch). Both pay the AU→US→Sydney geometry on every click. **The single observation I cannot make myself: does the operator perceive the lag as uniform-every-click, or spiky-first-click? (§9)** — this only adjusts how much weight cold start gets; it does not change the dominant cause.

- **Is it worst on navigation vs in-place actions?** Expected ordering, from the code:
  - **Portal in-session logging (`/portal/session/[dayId]`)** — each "Log set" tap = one Sydney RPC the user *waits on* with the button disabled (no optimistic update; `Logger.tsx` uses `useTransition` + `disabled={pending}`, not `useOptimistic`). The per-set path does **not** `revalidatePath`, so it is round-trip-bound (~one US↔Sydney RPC), not re-render-bound. Predicted: a consistent sub-second-to-second hitch on every set logged.
  - **Session builder (`/clients/[id]/program/days/[dayId]`)** — the heaviest surface: a US server action + Sydney RPC **+ `revalidatePath`** (this route's action file has the most revalidations of any, 21) → full server re-render with its Sydney queries, on every edit. Plus the largest client bundle (`@dnd-kit`). Predicted worst-feeling surface.
  - **Dashboard (`/dashboard`)** — navigation cost: middleware `getUser` (Sydney) + 10 queries batched into 2 parallel waves (well-built) + render, all from the US. Predicted heavy on load, lighter than portal because the queries are batched.

So on the surfaces the client touches most, the round-trip latency from §1 is felt directly with no instant feedback to mask it.

## 6. Step 3 — Server-side cost (round-trip and query time)

- **There is no server-action duration log to read** (see §2). The measurable server-side signals are the production TTFB probes and `pg_stat_statements`.
- **Production TTFB probes (warm, from AU):** no-DB function floor `/login` **0.14–0.25 s**; function-plus-one-Sydney-query `/i/<uuid>` **0.79–1.0 s** (1.7 s colder). The **~0.6–0.85 s delta is the US→Sydney query tax** for a *light* query path. This is the core server-side number of the whole pass.
- **`pg_stat_statements` — production query execution is NOT the bottleneck.** The view is heavily polluted by the pgTAP **test suite** (the top-mean rows are all `-- §N. Fixture` blocks, `DO $$ … v_org …` setup, and the `with f as (…)` pgTAP function-introspection CTE — test-harness queries, not user traffic). Filtering those out, the only production-relevant slow statement is **`SELECT name FROM pg_timezone_names`** (481 calls, 612 ms mean) — and that is the `organizations.timezone` **validation trigger** (`migrations/20260420100200_identity_tables.sql:56`), which fires on org create/update (rare in production; the 481 calls are mostly test fixtures creating orgs). **It is a write-time/test cost, not a read hot path, and irrelevant to interaction lag.** No production *read* query is slow — consistent with the brief's prior that at friends-and-family data volume the tables are tiny.
- **Why the round-trips stack — the architecture that amplifies the region tax:**
  - **`force-dynamic` is on essentially every route (~35 of them).** Nothing is cached or static; every navigation is a full SSR with live Sydney queries.
  - **Middleware re-verifies auth against Sydney on every single request.** `src/lib/supabase/middleware.ts:150` calls `await supabase.auth.getUser()` (network verify against the GoTrue auth server in Sydney) before the page even renders — on every navigation and every server action. Pages then call `getUser()` again themselves (e.g. `portal/page.tsx:40`, dashboard via `requireRole`), so a single navigation can pay two serial auth round-trips before any data query.
  - **Per-query HTTPS to Sydney.** `createSupabaseServerClient` (`server.ts`) talks to the Supabase REST endpoint; each `.from()/.rpc()` is a separate HTTPS round-trip from the `iad1` function to Sydney.
  - **Sequential awaits on the hottest read path.** `portal/page.tsx` runs `getUser` (40) → client row (43) → `resolvePortalToday` (54) → active programs (64) → `client_get_week_overview` RPC (154) → sessions (179) **in series** — ~5–6 cross-Pacific round-trips one after another. (The dashboard, by contrast, batches its 10 queries into two `Promise.all` waves — good practice that limits it to ~2–3 serial waves.)
  - **Mutations `revalidatePath` pervasively** (212 occurrences across 33 action files). A mutation awaits the Sydney RPC, then `revalidatePath` re-renders the whole route — re-running its Sydney queries — before the optimism-free UI updates. Laggy on every such click by construction.

## 7. Step 4 — Classification and the dominant bottleneck

Ranked against the four candidate classes:

1. **Per-interaction server round-trip — DOMINANT.** Measured: compute in `iad1`, DB in Sydney, users in AU; warm DB-touching TTFB ~0.8–1.0 s vs ~0.15 s no-DB floor; ~5–6 sequential such round-trips on the hottest authed surface. The root cause is the **infrastructure region mismatch**, and it manifests as a per-interaction round-trip cost. This singles itself out over the other three: it is present on **warm** requests (rules out cold-start-as-cause), it is server-side (the no-DB function floor is only ~0.15 s, but adding Sydney queries multiplies it — rules out client TTI as the driver), and DB *execution* time is low (rules out query cost).

2. **Server render / architecture — STRONG SECONDARY AMPLIFIER (not the root).** `force-dynamic` everywhere + double `getUser` + sequential awaits + pervasive `revalidatePath` + no optimistic UI turn one region-tax into *many* per interaction. This is what makes the dominant cause hurt as much as it does. After the region is fixed it becomes the next ceiling (each round-trip drops to ~2 ms, so 6 of them is ~12 ms instead of ~1.2 s — but batching, caching, and optimistic UI would still help).

3. **Server query cost — RULED OUT as dominant.** `pg_stat_statements` shows no slow production read; tables are tiny at f&f volume; the one slow statement is a write-time/test validation trigger.

4. **Infrastructure cold start — MINOR / ADDITIVE, partly unconfirmed.** A real Vercel cold-start penalty exists (apex cold 2.19 s vs warm 0.16 s), and a free-tier Supabase pause would add a large *first-after-idle* spike. But warm latency is already ~0.8–1.0 s, so cold start is additive on top of the dominant cause, not the cause itself. Final sizing needs the operator's idle-vs-uniform observation and the Supabase tier (§9).

**Two classes contribute materially: #1 (root) and #2 (amplifier), and they are coupled — #2 multiplies #1.** #1 is unambiguously the one to fix first.

## 8. Recommended smallest targeted fix (NOT implemented this pass)

**Move the Vercel serverless/SSR functions to the Sydney region (`syd1`).** This co-locates compute with both the Sydney database and the AU users, collapsing the two latency sources the measurements isolate (the AU→US user hop and the US→Sydney per-query hop) to near-LAN times. It is the smallest change that addresses the root cause: **no application code changes**, just deployment region configuration (Vercel project Functions region, or a `vercel.json` `"regions": ["syd1"]`).

Prerequisites and caveats (to be handled when the fix is actually designed, after review):
- Requires a **redeploy** — so not today (Friday change-freeze), and out of scope for this measure-only pass.
- Requires a **Vercel plan that allows region selection** (confirm — §9). If the project is on a plan that pins the default region, the plan change is the fix's prerequisite.
- Edge **middleware** already runs near the user; only the serverless functions need to move.

**Deliberately deferred to later passes (do not bundle into the region fix):** the §6 architectural amplifiers — collapse the double `getUser`, parallelise the portal's sequential awaits, add optimistic UI to portal logging and the session builder, and reconsider blanket `force-dynamic`/`revalidatePath`. These are the *next* ceiling after region, not this pass's fix, and each deserves its own measured before/after.

## 11. Pass 2 — syd1 region fix: preview proof (2026-06-26)

Config-only change on branch `perf/vercel-region-syd1` (commit `3ff235e`): a root `vercel.json` with `{ "regions": ["syd1"] }`, pinning serverless functions to Sydney (`syd1` = `ap-southeast-2`), co-locating compute with the database. No app code, middleware, migrations, or RLS touched. Proven on a Vercel **Preview** deployment (Ready); **not promoted to production**.

**Region flip — confirmed.** `x-vercel-id` on the preview reads **`iad1::syd1::…`** (compute now Sydney) vs production's **`iad1::iad1::…`**. The leading `iad1` is just the ingress PoP nearest the prober (operator's US machine); the compute segment is the headline, and it moved to `syd1`.

**DB tax collapse — the proof.** Same two routes; delta = `TTFB(DB-touching /i/<uuid>) − TTFB(no-DB /login)`:

| | no-DB `/login` | DB-touching `/i/<uuid>` | **Delta (cross-region DB tax)** |
|---|---|---|---|
| **Before** — prod, `iad1` compute | ~0.13 s | ~0.88 s (warm) | **~0.75 s** |
| **After** — preview, `syd1` compute | ~0.59 s | ~0.60 s | **~0.00 s** |

Paired back-to-back deltas after the fix: `+0.017 / +0.090 / +0.053 / −0.073 / −0.028 s` (one `+0.531 s` jitter outlier excluded — a single slow `/i` request the others don't corroborate). The tax fell from a **rock-solid ~0.75 s to zero-within-noise** — several paired deltas are *negative*, i.e. the DB-touching route is statistically indistinguishable from the no-DB route. The `syd1` function now reaches the Sydney DB locally (~2–5 ms/query) instead of across the Pacific (~200 ms/query).

**Why absolute `/login` rose (~0.13 s → ~0.59 s) and why it doesn't matter.** These probes ran from the operator's **US** machine; moving compute to Sydney moved it ~13,000 km *away from the prober*, so the requester→compute hop grew. That penalty is the **prober's alone, while travelling** — and it cancels out of the delta (both routes pay it equally). For the **AU audience (and the operator once home)** that hop becomes local too (~10–30 ms), so they win on the absolute as well. Per the brief, the delta is the headline, not the absolute-from-here.

**Extrapolation.** The portal home does ~5–6 sequential DB round-trips: ~1.0–1.2 s of pure DB tax before, ~10–30 ms after. Pass 1's dominant cause is removed for the AU audience.

### Step 5 — residual, named not fixed
The proven route (`/i/<uuid>`) is **unauthenticated**, so it does not exercise the authenticated amplifiers, and the **authenticated residual was not directly measurable on the preview** (no app login session; the Vercel bypass clears Vercel's wall, not the app's own auth). Named for a later pass (per the brief + Pass 1 §6):
- **Edge middleware is not pinned by `vercel.json regions`** — Vercel runs Routing Middleware in all regions by default, **fewer on Hobby** (Vercel docs, confirmed). So on an authenticated route the middleware may still execute outside `syd1`.
- **Middleware re-verifies auth against the Sydney auth server on every request** (`src/lib/supabase/middleware.ts:150`), plus the page's second `getUser` and the portal's sequential awaits.

**Not addressed this pass, not bundled.** Re-trigger: probe an authenticated DB route (e.g. `/portal`) against a `syd1` deployment with a real session; a few-hundred-ms residual there is the middleware path.
