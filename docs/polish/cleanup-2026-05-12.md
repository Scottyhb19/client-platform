## Cleanup mechanics

One `DO $$ … $$` transaction in the SQL Editor:

1. `ALTER TABLE user_organization_roles DISABLE TRIGGER enforce_last_owner_invariant;` — surgical disable of the last-owner-of-org invariant. The transaction restores it at the end; if the transaction had rolled back, the DDL would also have rolled back (verified during one failed attempt mid-session). FKs stayed enforced throughout — only this one trigger was off.
2. Deletes in leaf→root FK order: set_logs → exercise_logs → sessions → appointment_reminders → appointments → … → clients → user_organization_roles → `auth.users` (cascades to user_profiles) → audit_log → organizations.
3. Audit_log deleted **after** all tenant deletes but **before** organizations, because (a) the tenant deletes write new audit rows via AFTER triggers, and (b) `audit_log.organization_id` is `ON DELETE RESTRICT` so leftover rows would block the org delete.
4. `RAISE NOTICE` after each step for spot-checking.

## Verification

Post-cleanup diagnostic returned all zeros for both org_ids across every tenant-owned table and confirmed the two `auth.users` rows were gone. `LIMIT 1 WHERE role='owner'` is now deterministic.

## What was NOT touched

- No production-shaped data — both orgs were fixture-only and never had a real client log in.
- No migration file was written. The cleanup was a one-shot operational SQL run, not a schema change. The brief explicitly allowed this: pre-launch cleanup needs no migration footprint.
- No code changes. The `LIMIT 1 WHERE role='owner'` callers don't need updating — they were already correct; they just had stale data to filter through.

## Operational notes (for future cleanups)

- The plpgsql `users` and `orgs` variable names shadow `auth.users` and any "orgs" CTE/alias. Always prefix locals with `v_` (memory: *plpgsql var shadowing column*).
- `note_template_fields` FK column is `template_id`, not `note_template_id`.
- The `enforce_last_owner_invariant` trigger is the only DELETE-fire trigger that blocks an org-removal cleanup. `appointments_client_field_lockdown` is BEFORE UPDATE and irrelevant for deletes.
- Surgical `DISABLE TRIGGER` is preferable to `SET session_replication_role = 'replica'` for this kind of cleanup: replica mode silences FK system triggers too, which removes the safety net that catches a bad delete order.
