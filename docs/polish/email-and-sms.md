## 0. Executive summary

Section 12 is **two unlike things wearing one section number**, and naming that honestly is the most important thing this document does.

**Part A — genuine polish (small, in the brief).** The existing email plumbing is sound: Resend is live and verified, four templates are well-built, reminders send reliably (the scheduling §9 close-out fixed and proved the live send path). The polish work is real but bounded: kill the **silent send-failure** paths (booking confirmation and message notification swallow errors today — a real user can book and never get a confirmation while the EP never knows), an honesty pass on **template tone**, **preference handling** clarity, and **SMS honest-stub** hardening (no activation). And one happy finding de-risks a chunk of it: the `communications` log table is **already fully security-perimetered** — RLS (staff-only, org-scoped, `sender_user_id = auth.uid()` on insert, deny-delete; [`rls…:1168-1185`](../../supabase/migrations/20260420102600_rls_enable_and_policies.sql)), an `audit_communications` trigger, a matching `audit_resolve_org_id` CASE branch ([`audit…:230`](../../supabase/migrations/20260420102300_audit_log_and_triggers.sql)), coverage-guard registration, and `body` in the audit redaction list. The Comms-log security foundation is **done and tested**; it has simply never been written to.

**Part B — a new feature with a production-grade infrastructure surface (large, an owner-approved deviation from the brief's literal model).** The operator's mail-icon directive — *send through the account holder's own mailbox, provider-agnostic* — is **not polish.** It is the biggest single item in the entire polish pass by infrastructure weight, because nothing like it exists yet:

- a **provider-abstracted "connected email account" layer** (Google/Gmail first; Microsoft Graph and generic SMTP+IMAP admissible later — the "alternative sources still work" requirement is an *architecture* requirement);
- an **OAuth flow** + **encrypted at-rest storage of refresh tokens** (a refresh token is the ability to *send email as the EP* — a new, sensitive secret surface that did not exist before);
- a hard **external dependency** (a Google Cloud OAuth client + consent screen) and a real **operational landmine**: Google issues refresh tokens that **expire after 7 days while the OAuth app is in "Testing" publishing status** — so without a deliberate publishing decision, the headline feature breaks weekly and may break *silently*.

This deviates from brief §6.7's literal *"emails sent via EP's email domain using Resend."* But it **preserves and arguably better fulfils the brief's stated intent** — *"personal email contact makes clients feel valued rather than processed by a system"* — because the email is now genuinely from the EP's own mailbox, with replies landing naturally in the EP's inbox. This is the same governance shape as the messaging deviation (section 10): an owner-approved, intent-preserving departure, recorded as a decision. **It is not waivable into "just ship it" — the token-security and OAuth-publishing decisions (§4 Q1/Q3) gate the build and must be resolved first.**

**The clean architectural split that reconciles everything:** keep **Resend** for all *transactional/system* email (invites, booking confirmations, reminders, message notifications) — brief-correct, no change, high-deliverability, no human in the loop. Add the **connected account** only for the EP's *personal compose-to-client* email (the mail icon). **Log both classes to `communications`** → render in a new **Comms tab** (the brief's §6.7 requirement, currently unmet). One log, two delivery channels, the right channel for each job.

**Recommendation on sequencing:** treat Part B as a gated sub-project. Resolve §4 Q1 (FROM address) and Q3 (OAuth publishing / token lifetime) **before** any code — they change what gets built. Land Part A's reliability + log-wiring first (it makes the Comms tab meaningful and is low-risk), then Part B behind its decisions.

**Audit method:** brief §6.7/§3/§9.1 read line-by-line; three parallel read-only audits (email infra; the mail-icon + Comms-tab surface; SMS scaffolding) with file:line evidence; the `communications` table read in full and cross-checked against the live RLS policy file, the audit resolver, and the coverage guard; `package.json` and all migrations swept for any connected-account/OAuth library or token table (none found); the connected-account approach cross-checked against the CLAUDE.md "Gmail Send-mail-as" deferral and brief §6.7 (tension surfaced in §4 Q1, not silently resolved).
