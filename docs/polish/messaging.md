## 0. Executive summary

The messaging subsystem is **already built and, for its happy path, genuinely well-made** — it was landed in late April 2026 (migrations `20260420`–`20260426`) well before the polish pass reached it. The protocol's caution applies here in full: *the existing code is largely correct, and this pass must not assume otherwise.* What works, works cleanly:

- **A sound data model.** One thread per `(organization_id, client_id)` with denormalised `last_message_*` for fast inbox rendering; `messages` rows with `sender_role` **captured at insert** (survives a staff demotion), an `organization_id` denormalised onto the hot table for join-free RLS, and a 1–1000-char body cap (`messages.sql:77`) deliberately keeping this lane for *short coordination, not clinical narrative*.
- **Both surfaces, fully wired.** A staff inbox (thread list, search, conversation pane, client snapshot side panel, quick replies, optimistic send with realtime canonicalisation) and a mobile portal thread (same realtime/optimistic core). Unread badges live-update in both navs via realtime.
- **Cross-tenant RLS present** on both tables for both roles, **hard DELETE denied** (soft-delete only — retention-aware per §7.2), the archive-cascade keeps a thread's `deleted_at` in lockstep with its client, and a `enforce_same_org_fk` trigger guards the thread→org and message→thread→org invariants.
- **A real clinical-safety instinct already in the UI** — a *"For urgent issues, call 000"* banner on the staff side panel (`Inbox.tsx:600-603`) and the portal (`ClientThread.tsx:217,318`), plus *"Reply within business hours."*

So this is **not** a build. The work is concentrated in three places, none of them the happy path:

1. **The compliance/security perimeter — the condition the owner's approval was predicated on.** The deviation was approved *because* messaging would stay "inside the existing RLS/audit perimeter … subject to APP compliance + DR tooling." Two halves of that condition are currently **unmet**: (a) the migration header claims messages are *"immutable except `read_at`/`deleted_at` — enforced via update RLS,"* but the actual UPDATE policies impose **no column-level restriction**, so within a thread a client can edit or soft-delete the **EP's** messages and forge attribution (FM-1); and (b) `messages`/`message_threads` carry **no audit trigger** and are **absent from `audit_resolve_org_id()`**, so that tampering is also **untraceable** (FM-2), against §7.4. Add the named anon-EXECUTE rider (FM-3) and the **complete absence of RLS test coverage** on a brand-new tenant surface (FM-4), and the four P0s are all perimeter, not features.

2. **The "texting-style feel" gap.** Texting *notifies*. There is **no out-of-app notification** of any kind — no email-to-EP on a new client message, no web push (FM-5). A friends-and-family tester who messages the EP gets silence until the EP next opens the inbox; the same in reverse. The unread badge only moves while the app is already open.

3. **A short tail of polish** — clinical-safety disclosure strength, a token sweep, compose-flow friction, and the external-review doc gap.

The audit method: deep reads of every messaging file (DB, both server actions, both client components, both navs) cross-checked against the audit-resolver coverage guard, the pgTAP inventory, and the go-live rider text. The two security findings (FM-1 immutability, FM-2 audit) were each verified twice — once against the creating migration, once against the resolver CASE list and the full migration set (`grep` for any later audit-attach on messages → none).
